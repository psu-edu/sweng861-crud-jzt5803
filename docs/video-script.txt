---SLIDE 2 — Requirements---

[00:30]

The claim here is that Campus Analytics eliminates university metrics data silos.

On the left, you can see the three specific problems it solves. There's no audit
trail — nobody knows who changed what or when. There's no access control —
anyone with the spreadsheet can edit any data. And there's no single source of
truth — every department is working from a different version of the file.

On the right is the evidence — seven features, all shipped. Secure login with
JWT and Google OAuth. A five-category metrics system. A live weather widget.
An audit trail that's append-only and can't be modified. Prometheus and Grafana
for observability. GitHub Actions for CI/CD. And a Docker image that runs
without root privileges.

The non-functional targets aren't just listed — they're implemented. OWASP Top
10 mitigations for security. A 99.5% availability target and p95 latency under
500ms for reliability. And 320 automated tests that block any broken build from
ever getting packaged.

So now you know what the app does and why it was built. Let's look at how it's
all put together architecturally.

[CLICK to next slide]


---SLIDE 3 — Architecture---

[01:30]

The claim is that Next.js 15 collapses the frontend and API into one single
deploy unit.

On the left, the four-line proof: one codebase, one Docker image, no CORS,
no split infrastructure. The diagram on the right shows what that looks like.
The browser, the fifteen API route handlers, the SQLite database, and the
DevOps stack all live inside one Next.js application. One build command
produces the entire thing.

Most web projects need two separate deployments — a React frontend and a
Node.js backend. With Next.js App Router, you get both in one. That's one
fewer image to build, one fewer server to scale, and zero CORS headers
to configure.

That architecture runs on a specific set of tools — and each one was picked
for a specific reason. That's the next slide.

[CLICK to next slide]


---SLIDE 4 — Technology Stack---

[02:25]

The claim here is that every technology choice was made to minimize overhead
and maximize testability.

The "Why" column gives a specific reason for each one. Let me call out a few.

SQLite and Sequelize: zero infrastructure to start. The database file gets
created automatically on first run. No database server, no setup step.

Open-Meteo for weather: it's free and keyless, so there are no API secrets
to manage, rotate, or accidentally commit.

Jest and React Testing Library: 320 tests run in CI. If any one of them
fails, the build stops right there — nothing gets packaged into Docker.

And Prometheus plus Grafana: a five-panel SLO dashboard, no external cloud
dependency. It runs in the same docker-compose stack as the app.

Now let's go deeper on what those tools actually deliver — the features
themselves.

[CLICK to next slide]


---SLIDE 5 — Core Features---

[03:25]

The claim is that multi-layer defense delivers seven production-grade features.

On the left, auth and security. JWT, NextAuth sessions, and Google OAuth all
go through one getAuthUser function — the route handlers don't care which
token type the client is using. Brute force is blocked at two levels: bcrypt
at cost 12 deliberately adds 200ms per hash check, and the rate limiter cuts
off after five auth attempts per fifteen-minute window per IP. HSTS, BOLA
owner checks, and sensitive key stripping from logs ship on every response.

On the right, the data features. Five server-enforced category enums, UUID
keys to prevent ID enumeration, US locale formatting everywhere, a ten-minute
weather cache so we're not hammering the external API, and an audit trail that
writes every mutation asynchronously — so it never slows down the API response.

All of those features need a reliable way to get built and deployed. That's
the next slide.

[CLICK to next slide]


---SLIDE 6 — DevOps and Observability---

[04:25]

The claim is that every commit is automatically built, tested, and packaged —
no manual steps.

The pipeline diagram on the left shows two jobs. Job one is build and test.
npm ci does a lockfile-exact install so it's reproducible every time.
npm run build compiles the Next.js output. Then npm run test with coverage
collection — and there's no continue-on-error flag, so one failing test stops
the pipeline right there. That's the quality gate. After that, npm audit
rejects anything with a high or critical CVE.

Job two only starts if job one passes. It builds the Docker image in three
stages — install, build, then a minimal Alpine runtime that has no source code
and no dev dependencies in it. The container runs as user nextjs with UID 1001,
never as root. Then a smoke test: it calls /api/health five times and if the
app doesn't respond, the pipeline fails.

On the right, three observability lines. Three health endpoints each with one
role. Structured JSON logs with sensitive keys stripped. Four Prometheus metrics
feeding a five-panel Grafana dashboard, with SLOs at 99.5% availability and
p95 latency under 500ms.

A lot of those DevOps files were drafted with AI help — and that's worth
being upfront about.

[CLICK to next slide]


---SLIDE 7 — AI Audit---

[05:40]

The claim is that AI-generated code had three security flaws — and all three
were caught by manual review.

Quick note on where AI actually helped: scaffolding the initial CI YAML, the
Dockerfile structure, and the Prometheus config all saved real time. I'd use
it again for those tasks. But every file it touched that involved infrastructure
or security got a manual review before I committed it. That review is what
found these three.

First: hard-coded JWT_SECRET in the YAML — credentials sitting in git history,
visible to anyone with repo access. Fixed by using a GitHub Secrets reference.

Second: continue-on-error true on the test step — failing tests would show
a green pipeline. That defeats the whole point of CI. Fixed by removing
the flag entirely.

Third: no USER directive in the Dockerfile — the process would run as root.
If any dependency had a remote code execution bug, an attacker would land
as root. Fixed by adding a non-root user before the CMD.

The lesson at the bottom of the table: AI optimizes for working code,
not for security defaults. Every AI output that touches infrastructure
needs a security pass.

The five lines below the table are separate technical challenges I hit during
development — each one was a root-cause investigation, not a quick fix.

Those flaws were caught by review — but you also need a solid testing strategy
to make sure broken code can't sneak through in the first place.

[CLICK to next slide]


---SLIDE 8 — Testing---

[07:00]

The claim is that 320 tests across three isolation layers form a CI quality gate.

The table has three columns. Column one: 196 unit tests covering auth,
validation, rate limiting, domain events, API errors, weather service, and
middleware. No external dependencies, no database, no HTTP calls. Fast.

Column two: 124 frontend component tests using React Testing Library in jsdom.
No browser needed. They cover every interactive component — login form,
metric form, navbar, weather widget, auth provider, and the API client.

Column three: integration tests. One file, api.test.js, using Node's built-in
test runner against a live server on port 3001. The full cycle — register,
login, CRUD, weather, domain events. These run locally because they need a
running server; they're not in CI.

The bottom right cell is the quality gate rule: if any test fails, the pipeline
stops. The Docker build stage never starts.

With that foundation in place, let me close with what I actually learned
from building all of this.

[CLICK to next slide]


---SLIDE 9 — Reflections---

[07:55]

The claim is that observability has to be designed in — not bolted on.

Left column: three things built from scratch. A zero-dependency Prometheus
module — 170 lines of plain Node.js, no external libraries. A unified auth
bridge so JWT, NextAuth, and OAuth are all transparent to the route handlers.
And two infrastructure bugs that only showed up in end-to-end QA, not in
any of the 320 tests.

Middle column: three lessons. Log structured JSON before you need it — adding
it after the fact would mean touching every route handler. AI drafts, humans
review security — the previous slide is the evidence. And end-to-end testing
finds what unit tests miss.

Right column: what breaks at scale. SQLite becomes PostgreSQL. The in-memory
rate limiter becomes Redis. Single-instance becomes Kubernetes.

Let me close with what I actually learned from this.

The hardest parts weren't the features — they were two infrastructure bugs that
no unit test caught. One was a one-word fix: sync instead of sync-with-alter.
The other was aligning three-level fallback chains across two files. Both bugs
were only findable by running the actual application end-to-end. That's the
best evidence for the assertion on the last slide.

What I'm most proud of: the zero-dependency Prometheus module, the 320-test
suite, and the multi-layer auth bridge that makes all three token types
invisible to the route handlers.

The patterns from this project — CI/CD, structured logging, health checks,
metrics dashboards — show up in every production system. The tools change.
The patterns don't. This project gave me a working implementation of all of
them, not just the concepts.
